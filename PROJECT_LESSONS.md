# 量化平台 V3 - 踩坑复盘与后续借鉴

本文档记录 V3 调试过程中已经踩过的坑、确认过的事实和后续开发建议。后续继续开发时，优先参考这里，避免重复排查。

## 一、网络与 OKX API

### 1. 官网/App 能用，不代表 REST API 能用

OKX 官网、OKX 桌面客户端、手机 App 可以正常登录和交易，不等于量化平台可以直连 OKX API。

量化平台通过 CCXT 调用的是官方 REST API：

```text
https://www.okx.com/api/v5/...
```

判断本机是否能直连 OKX API，只看这个接口：

```text
https://www.okx.com/api/v5/public/time
```

如果不开代理打不开这个地址，则：

- Public API 不通
- 余额接口不通
- 持仓接口不通
- 模拟盘接口不通
- 下单接口也不通

不要用“官网能打开”或“OKX App 能登录”判断 API 是否可用。

### 2. 当前网络下 OKX REST API 必须走代理

实测当前环境中，OKX REST API 不开代理不可访问。开代理后，通过本机代理端口可连接成功。

已验证成功路径：

```text
127.0.0.1:PORT
```

平台当前策略应为：

```text
OKX API: 自动代理优先 -> 直连/备用域名
```

不要再把“直连优先”作为 OKX 的主路径，否则会浪费时间并造成实盘监控无响应。

### 3. DNS 可能指向异常地址

曾经检测到：

```text
www.okx.com -> awscn.okpool.top -> 169.254.0.2
```

`169.254.*` 是链路本地地址，不是 OKX 正常公网 API 地址。出现这种解析结果时，REST API 必然不可用。

对比公共 DNS：

```text
1.1.1.1 -> www.okx.com.cdn.cloudflare.net -> Cloudflare 公网 IP
```

因此 API 问题排查顺序应为：

```text
DNS -> TCP 443 -> public/time -> public ticker -> private balance -> sandbox header
```

不要一开始就怀疑 API Key。

### 4. 镜像站不能发送 API Key

以下域名可以作为网页登录入口，但不应作为 OKX 私有 API 请求目标：

```text
cnoyu.me
ouyi.com.im
```

原因：

- 它们更像前端镜像/网页登录入口
- 不一定开放 `/api/v5`
- DNS 链路中出现多级第三方域名
- 不应向这些域名发送 API Key、Secret、Passphrase

私有 API 只能发送到可信的 OKX 官方 API 域名或 CCXT 官方支持的 OKX host。

### 5. OKX 客户端和 API 是两条链路

OKX 桌面客户端可能使用：

- 内置线路选择
- App 专用网关
- 长连接
- 本地服务端口
- 前端 CDN
- 地区化入口

这些链路不能直接复用到 CCXT。量化平台应坚持走官方 REST/WebSocket API，不要尝试模拟客户端私有接口。

## 二、DeepSeek / AI 助手

### 1. DeepSeek 也受网络路径影响

当前环境中，DeepSeek 直连可能失败，但通过本机代理 `127.0.0.1:PORT` 可成功。

AI 连接失败时，优先检查：

```text
本机代理端口是否可用
api.deepseek.com 是否能访问
OpenAI SDK 是否安装
API Key 是否存在
模型名是否正确
```

不要只看页面显示“AI超时”就判断 Key 错。

### 2. 市场分类不要直接依赖推理模型

保存的模型可能是：

```text
deepseek-v4-pro
```

这类推理模型可能先返回 reasoning 内容，导致 `content` 为空或 JSON 不稳定。

市场状态分类应优先使用结构化输出更稳定的模型：

```text
deepseek-chat
```

AI 市场状态输出必须限制为：

```text
TRENDING_UP
TRENDING_DOWN
RANGING
HIGH_VOLATILITY
趋势末端
```

返回格式异常、空返回、超时时，必须自动降级为本地规则，不能一直卡在“分析中”。

### 3. AI 只做建议，不能修改风控

AI 可以做：

- 市场状态判断
- 策略适配理由
- 参数优化建议
- 回测复盘解释

AI 不能修改：

- 总杠杆限制
- 单策略风险
- 冷却期
- 强制平仓规则
- 趋势末端处理规则
- API 密钥和交易权限

风控必须硬编码。

## 三、策略库与组合方式

### 1. 策略库不是让 AI 随便挑一个

当前策略库中有 7 个质检策略：

```text
SMC_结构转换突破_ETH
趋势回调入场
BOS移动止损增强版
趋势衰竭反转
摆动点区间反转
BB挤压突破
成交量异动
```

还有 2 个协同策略：

```text
横截面动量选币
资金费率反转
```

正确设计不是“AI 从 7 个里面选一个”，而是：

```text
AI 判断市场状态
硬编码调度器决定策略组合
风控模块决定仓位和是否允许执行
```

### 2. 推荐策略矩阵

趋势市：

```text
趋势回调入场
BOS移动止损增强版
SMC_结构转换突破_ETH（仅 ETH 专用增强）
```

震荡市：

```text
摆动点区间反转
```

高波动市：

```text
BB挤压突破
成交量异动
```

趋势末端：

```text
趋势衰竭反转
```

非方向性套利：

```text
资金费率反转
```

选币层：

```text
横截面动量选币
```

### 3. 调度规则建议

```text
TRENDING_UP / TRENDING_DOWN:
  激活 趋势回调入场 + BOS移动止损增强版
  ETH 可观察 SMC_结构转换突破_ETH

RANGING:
  只激活 摆动点区间反转
  禁用趋势和突破策略

HIGH_VOLATILITY:
  激活 BB挤压突破 + 成交量异动
  双策略同向时各 0.5% 风险

TREND_END / 趋势末端:
  强制平掉顺势仓位
  只允许 趋势衰竭反转

ALL:
  资金费率反转独立评估
  横截面动量选币负责标的权重
```

### 4. 仓位与风控建议

```text
单策略: 1.0%
双策略共振: 每个 0.5%
趋势衰竭反转: 1.5%
BB挤压突破: 1.0%
成交量异动: 1.0%
资金费率反转: 独立 1.0%
总杠杆: <= 3x
策略切换冷却期: 12 小时
```

趋势末端优先级最高，触发时应无视冷却期，先处理顺势仓位。

## 四、参数与质检

### 1. 参数包要合并，不要只取增量字段

部分策略的 BTC/ETH/SOL 参数包是完整参数，部分是校准补丁。

例如：

```text
BOS BTC 可能只覆盖 n_swings=1
摆动点 SOL 可能只覆盖 rr_ratio/adx_threshold
```

正确方式：

```text
默认参数 -> 标的参数 -> 质检校准参数 -> AI 建议参数
```

AI 建议参数必须限制在原参数合理范围内，且不能覆盖风控字段。

### 2. 标的过滤必须硬编码

示例：

```text
SMC_结构转换突破_ETH 仅 ETH
趋势回调入场 不推荐 SOL
BOS ETH 主力，SOL 可用，BTC 谨慎
成交量异动 ETH/SOL 主力，BTC 不推荐
趋势衰竭 BTC/ETH 主力，SOL 不推荐
```

不要因为参数能读到，就让策略在所有标的上运行。

## 五、历史重放与回测

### 1. 历史重放不是让 AI 创造收益

AI 的作用是市场状态识别和策略调度，不是替代策略本身。

正收益来自：

- 策略本身有效
- 参数适配
- 市场状态匹配
- 风控执行
- 交易成本控制

不能期待 AI 多判断几次就自然产生正收益。

### 2. 回测应按市场状态分段

不要拿趋势策略在全市场硬测后直接判断好坏。更合理的报告应包括：

```text
趋势段表现
震荡段表现
高波动段表现
趋势末端表现
各策略独立表现
策略组合表现
策略启用/禁用原因
```

历史重放报告应显示：

```text
市场状态
激活策略
观察策略
禁用策略
参数包
动作
盈亏
决策质量
```

## 六、云端部署

### 1. 国内轻量云不适合直接跑 OKX API

国内轻量云大概率会遇到同样问题：

```text
www.okx.com/api/v5/public/time 不稳定或不可访问
```

可以用于：

- 回测
- 报表
- 数据库
- 策略研究
- 监控面板

不建议用于直接下单。

### 2. 云端交易执行器推荐地区

优先：

```text
香港
新加坡
日本
美国
```

上线前必须测试：

```bash
curl https://www.okx.com/api/v5/public/time
```

能长期稳定返回，才适合部署交易执行器。

### 3. 推荐架构

```text
本地桌面端:
  回测、AI分析、策略调参、监控

云端 Bot:
  行情、策略调度、风控、下单、日志、告警

可选中继:
  本地桌面 -> 自有云端 API -> OKX
```

云端不应依赖本机代理。

## 七、后续开发优先级

1. 统一网络诊断面板
2. OKX/DeepSeek 连接状态独立显示
3. 策略矩阵调度器
4. 参数合并器
5. 标的适配过滤器
6. 历史重放加速与分段报告
7. AI 市场状态解释与本地兜底透明化
8. Headless 云端交易执行器
9. 日志、告警和断线熔断
10. 云端模拟盘长期测试

## 八、一句话原则

V3 不应是“AI 选策略平台”，而应是：

```text
AI 判断市场
策略库分工协作
硬编码风控调度
云端稳定执行
本地桌面监控和复盘
```
