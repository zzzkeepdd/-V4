## 一、网络与交易所 API 对接

### 1.1 OKX API 直连不可靠——DNS 动态劫持

**坑**：OKX 的 AWS 中国 CDN（`awscn.okpool.top`）对国内用户做动态 DNS 解析，不同时间返回不同 IP。有时给真实可达 IP（直连 0.5s 通），有时给 `169.254.0.2`（链路本地死地址，100% 不可达）。

**诊断方法**：
```python
import socket
print(socket.gethostbyname('www.okx.com'))
# 如果返回 169.254.x.x → 死地址，直连不可能成功

# 用 DoH 验证是否上游 DNS 问题
import requests
r = requests.get('https://doh.pub/dns-query', 
    params={'name': 'www.okx.com', 'type': 'A'},
    headers={'accept': 'application/dns-json'})
print(r.json())
```

**经验**：
- 不要假设"之前能连现在也能连"——DNS 返回值随时可能变
- 国内稳定连 OKX API **必须走代理**，直连是薛定谔的
- `www.okx.me` 备用域名解析到真实 IP 但 SSL 证书不匹配，不可用
- 设计连接逻辑时必须实现**代理降级重试**（直连→备用域名→代理）

### 1.2 CCXT 4.x 不会自动读系统代理——但 aiohttp 会偷读 Windows 注册表

**坑**：CCXT 4.x 底层用 `aiohttp`，`aiohttp` 在 Windows 上会读 IE 系统代理设置（注册表 `HKCU\Software\Microsoft\Windows\CurrentVersion\Internet Settings` 中的 `ProxyEnable` + `ProxyServer`）。即使代理软件已关闭、端口无人监听，aiohttp 仍然尝试连接死端口导致超时。

**已证明无效的方法**：
- `proxies: {}` → ❌ aiohttp 忽略此参数仍读注册表
- `os.environ["NO_PROXY"] = "*"` → ❌ 仅影响环境变量，不覆盖注册表
- `os.environ["HTTP_PROXY"] = ""` → ❌ 同上
- 猴子补丁 `urllib.request.getproxies = lambda: {}` → ❌ aiohttp 有独立代理检测路径
- 手动 `reg add` 设置 `ProxyEnable=0` → ❌ 有软件（如 Clash）在后台反复写回注册表

**有效方法**：
- `requests.Session(trust_env=False)` → ✅ 绕过系统代理，但仅适用于 requests 库
- 启动代理软件让端口真正在监听 → ✅ CCXT/aiohttp 走代理成功
- **在 import ccxt 前**，用 `subprocess` 调用 `reg add` 关闭代理 + 清除 `ProxyServer` 值 → ⚠️ 可能被后台软件覆盖

**经验**：
- CCXT 的 `proxies` 参数只对部分传输层生效，aiohttp 有自己的代理发现机制
- Windows 上代理问题排查优先看注册表，不要只看环境变量
- `urllib.request.getproxies()` 能看到当前 Python 进程实际使用的代理来源

### 1.3 CCXT 异常检测必须用 isinstance，不能字符串匹配

**坑**：CCXT 抛出的异常类在模块嵌套中，用 `str(e)` 匹配错误文本不可靠。不同版本异常消息格式可能变。

```python
# ❌ 错误做法
if "RequestTimeout" in str(e):
    ...

# ✅ 正确做法
import ccxt
if isinstance(e, (ccxt.NetworkError, ccxt.RequestTimeout)):
    ...
```

### 1.4 CCXT 的 `urls` 是嵌套字典，不能传字符串

**坑**：尝试替换 OKX API URL 时，`exchange.urls` 是嵌套结构如 `{'api': {'public': '...', 'private': '...'}}`，不能直接赋值字符串。

```python
# ❌ 错误
exchange.urls['api'] = 'https://www.okx.com'  # 破坏嵌套结构

# ✅ 正确：递归替换域名
def _replace_url_domain(urls, old_domain, new_domain):
    if isinstance(urls, dict):
        return {k: _replace_url_domain(v, old_domain, new_domain) for k, v in urls.items()}
    elif isinstance(urls, str):
        return urls.replace(old_domain, new_domain)
    return urls
```

## 二、Streamlit 框架

### 2.1 Widget Key 冲突——页面路由完全阻塞

**坑**：在 `st.text_area(key="template_text")` 控件创建后，又用 `st.session_state.template_text = value` 对同一键赋值，触发 `StreamlitAPIException`。如果这个赋值在 `main()` 中部，会阻止下方所有 `if-elif` 页面路由逻辑执行，导致**任何导航点击均无响应**。

**症状**：侧边栏能点击但页面不变，不报错或报错被吞。

**解决**：永远不要在控件创建后对同一 key 做 `session_state` 赋值。如果需要初始化默认值，在控件创建前用 `if key not in st.session_state` 守护。

```python
# ❌ 错误——控件后赋值冲突
st.text_area("模板", key="template_text", value=template_text)
st.session_state.template_text = template_text  # 💥 冲突！

# ✅ 正确——控件前初始化
if "template_text" not in st.session_state:
    st.session_state.template_text = default_text
st.text_area("模板", key="template_text")
```

### 2.2 Streamlit Auto-Reload 缓存——代码改了但运行的是旧版本

**坑**：Streamlit 检测到文件修改后自动 reload，但对**结构性变更**（如函数从闭包移至模块顶层、函数签名修改）可能不会正确重新加载，旧代码仍在内存中执行。

**更坑**：`__pycache__` 里的 `.pyc` 文件也会缓存旧编译结果。

**解决**：
1. 修改代码后，必须 **Ctrl+C 彻底终止** Streamlit 进程
2. 删除 `__pycache__` 目录下的 `.pyc` 文件
3. 重新启动 `streamlit run`

```powershell
# 清缓存 + 重启
Remove-Item -Recurse -Force D:\量化平台\__pycache__
streamlit run D:\量化平台\monitor.py --server.port 8501
```

### 2.3 函数从闭包移至模块顶层——参数名不要用下划线前缀

**坑**：函数从 `main()` 内部闭包移到模块顶层后，参数名带下划线前缀（如 `_testnet`）可能与 pytest 的内部注入变量 `_pytest` 冲突，导致 `UnboundLocalError`。

**解决**：模块级函数不用下划线前缀参数名，尤其避免 `_test`、`_pytest` 等模式。

### 2.4 Streamlit 首次启动邮箱引导卡住服务

**坑**：Streamlit 首次启动会弹出邮箱收集引导，在某些环境下阻塞服务启动。

**解决**：启动命令加参数跳过：
```powershell
streamlit run monitor.py --browser.gatherUsageStats false
```
或在首次启动时直接回车跳过。

## 三、Python 数据处理

### 3.1 JSON 序列化遇 NaN/Inf/Decimal 直接崩溃

**坑**：`json.dump()` 遇到 `float('nan')`、`float('inf')`、`Decimal()` 直接抛 `TypeError` 或 `ValueError`。回测结果中经常出现这些值（除零、无穷收益率等）。

**解决**：写一个安全序列化函数：
```python
import numpy as np
from decimal import Decimal

def _json_safe(obj):
    """递归清洗 JSON 不可序列化的值"""
    if isinstance(obj, float):
        if np.isnan(obj): return None
        if np.isinf(obj): return "Infinity" if obj > 0 else "-Infinity"
        return obj
    if isinstance(obj, (np.integer,)): return int(obj)
    if isinstance(obj, (np.floating,)): return _json_safe(float(obj))
    if isinstance(obj, Decimal): return float(obj)
    if isinstance(obj, dict): return {k: _json_safe(v) for k, v in obj.items()}
    if isinstance(obj, (list, tuple)): return [_json_safe(i) for i in obj]
    return obj
```

### 3.2 配置文件保存——必须用合并模式而非覆盖

**坑**：`save_config()` 直接 `json.dump()` 覆盖文件，会丢失其他字段（如测试写入的临时字段、其他模块保存的凭证）。

**解决**：
```python
import json, pathlib

def save_config(updates: dict):
    config_file = pathlib.Path("config.json")
    existing = {}
    if config_file.exists():
        try:
            existing = json.loads(config_file.read_text(encoding="utf-8"))
        except: pass
    existing.update(updates)
    config_file.write_text(json.dumps(existing, ensure_ascii=False, indent=2), encoding="utf-8")
```

### 3.3 save_config 必须包裹 try/except

**坑**：配置保存失败时静默崩溃，用户以为保存成功了，下次启动发现凭证丢失。

```python
def save_config(updates: dict):
    try:
        # ... 合并 + 写入逻辑
    except Exception as e:
        print(f"配置保存失败: {e}")
        # 或 logging/st.error 等用户可见提示
```

## 四、回测引擎

### 4.1 信号重用——平仓后旧信号被反复执行（5 万倍虚增收益）

**坑**：BOS 策略中，平仓后 `position` 归零，但旧的回调信号仍满足 `pullback_idx < i` 条件，同一信号被反复执行。每次用更大本金复利增长，1 万本金变 5.68 亿。

**诊断特征**：回测收益率离谱（几万倍~几十万倍），交易次数异常多。

**解决**：新增 `last_exit_idx` 追踪上次平仓位置，入场前检查：
```python
last_exit_idx = -1  # 初始化

# 平仓后
last_exit_idx = i

# 入场前
if i <= last_exit_idx:
    continue  # 跳过已用过的信号
```

### 4.2 入场价回溯——未来数据泄露（前视偏差）

**坑**：`entry_price = close[pullback_idx]` 使用过去 K 线收盘价作为入场价，但实际入场应发生在当前 K 线。

**解决**：
```python
# ❌ 错误——用过去K线价格入场
entry_price = close[pullback_idx]

# ✅ 正确——用当前K线价格入场
entry_price = close[i]
```

### 4.3 模拟数据回退——价格不锚定真实市场

**坑**：API 连接失败回退模拟数据时，模拟 OHLCV 起点与真实市场价格差距巨大，导致回测结果毫无参考价值。

**解决**：维护全局 `_LAST_REAL_CLOSE` 价格锚点，模拟数据从真实收盘价起步：
```python
_LAST_REAL_CLOSE = None  # 模块级全局变量

def _generate_simulated_ohlcv(symbol, timeframe, since, limit):
    start_price = _LAST_REAL_CLOSE or 50000  # 有锚点用锚点，无则用默认
    # 生成数据时累计涨跌不超过 10 倍，年化波动率 ≤ 150%
```

### 4.4 最小仓位保护——低于交易所最小下单额

**坑**：回测中仓位价值低于交易所最小下单额（OKX 最低约 $10），实际交易会被拒绝，但回测引擎照算收益。

**解决**：
```python
MIN_POSITION_VALUE = 10.0  # 美元

def validate_position_size(position_value):
    return position_value >= MIN_POSITION_VALUE

# 回测后过滤
skipped = sum(1 for t in trades if not validate_position_size(t['value']))
```

### 4.5 回测报告必须带时间区间描述

**坑**：只写起止日期不够直观（"2025-01-01 至 2025-12-31"），应注明"13个月"或"1年"。

## 五、文件与部署安全

### 5.1 文件被意外覆盖——2700 行代码瞬间变 500 行

**坑**：一次误操作（可能是 AI 辅助编辑时选错了目标段），整个 `monitor.py` 从 2700 行/92KB 被覆盖为 519 行/14KB 的 CLI 脚本。

**惨痛教训**：
- **必须用 Git**。没有版本控制 = 裸奔
- 重要项目在桌面/其他位置保留备份
- AI 辅助编辑大文件时，先用 `read` 确认编辑范围，不要盲信 `edit` 工具的匹配

**恢复路径**：
```powershell
# 从桌面备份恢复
Copy-Item "C:\Users\Administrator\Desktop\量化平台代码\monitor.py.txt" "D:\量化平台\monitor.py"
# 然后逐个重新应用修复补丁
```

### 5.2 测试污染真实配置文件

**坑**：验收测试直接调用 `save_config({"test_key": "test_value"})`，写入真实 `config.json`，导致生产配置被污染。

**解决**：测试必须隔离文件系统：
```python
import pathlib, tempfile

def test_save_config():
    with tempfile.TemporaryDirectory() as tmp:
        config_file = pathlib.Path(tmp) / "config.json"
```

## 六、策略开发

### 6.1 ATR 止损倍数必须按品种调整

**坑**：BTC 和 ETH 波动率差异大。BTC 用 `stop_atr_mult=1.0` 合理，ETH 用同样的 1.0 导致止损过紧，80% 回撤。

**经验**：ETH 波动率更高，ATR 止损倍数通常需要 1.5x 甚至 2.0x。

### 6.2 ADX 滞后期假信号——需要趋势一致性过滤器

**坑**：ADX 指标滞后，在趋势转向初期仍显示高值，导致 BOS 策略在假突破处入场。

**解决**：稳健版策略新增趋势一致性过滤器——做多需同时满足 `+DI > -DI` 且 `EMA5 > EMA20`，防止 ADX 滞后期的假信号。

### 6.3 策略文件必须声明参数结构

**坑**：AI 参数优化模块无法知道策略有哪些可调参数，只能瞎调 leverage。

**解决**：策略文件头部声明 `PARAMS_SCHEMA`：
```python
PARAMS_BOS_DEFAULT = {
    "swing_lookback": 20,
    "n_swings": 3,
    "sl_buffer": 0.001,
    "bos_buffer": 0.0005,
    "tp_rr": 2.0,
    "vol_filter": True,
    "vol_lookback": 20,
    "vol_mult": 1.5,
}

PARAMS_SCHEMA = {
    "swing_lookback": {"min": 5, "max": 50, "step": 1},
    "n_swings": {"min": 2, "max": 5, "step": 1},
    "tp_rr": {"min": 1.0, "max": 5.0, "step": 0.5},
    # ...
}
```

## 七、Windows 环境特有问题

### 7.1 Windows 系统代理注册表位置

```
HKCU\Software\Microsoft\Windows\CurrentVersion\Internet Settings
  ProxyEnable   REG_DWORD  0x0=关  0x1=开
  ProxyServer   REG_SZ     127.0.0.1:7897
```

**查看/修改**：
```powershell
# 查看
reg query "HKCU\Software\Microsoft\Windows\CurrentVersion\Internet Settings" /v ProxyEnable
reg query "HKCU\Software\Microsoft\Windows\CurrentVersion\Internet Settings" /v ProxyServer

# 关闭代理
reg add "HKCU\Software\Microsoft\Windows\CurrentVersion\Internet Settings" /v ProxyEnable /t REG_DWORD /d 0 /f

# 清除代理地址
reg delete "HKCU\Software\Microsoft\Windows\CurrentVersion\Internet Settings" /v ProxyServer /f
```

**注意**：Clash 等代理软件会持续监控并写回注册表，手动改可能被覆盖。

### 7.2 自动检测系统代理的代码

```python
import winreg

def _detect_system_proxy():
    """读取 Windows IE 系统代理设置"""
    try:
        key = winreg.OpenKey(winreg.HKEY_CURRENT_USER,
            r"Software\Microsoft\Windows\CurrentVersion\Internet Settings")
        enabled, _ = winreg.QueryValueEx(key, "ProxyEnable")
        server, _ = winreg.QueryValueEx(key, "ProxyServer")
        winreg.CloseKey(key)
        if enabled and server:
            return {"http": f"http://{server}", "https": f"http://{server}"}
    except:
        pass
    return {}
```

## 八、排查方法论总结

### 8.1 网络连通性分层排查

```
L1: ping 域名 → DNS 解析是否正常？
L2: nslookup 域名 8.8.8.8 → 是否本地 DNS 劫持？
L3: python requests 直连 → HTTP 层是否通？
L4: python ccxt 连接 → 库层面是否正常？
L5: 检查 urllib.request.getproxies() → 系统代理是否干扰？
L6: 检查注册表 ProxyEnable/ProxyServer → Windows 代理残留？
```

### 8.2 Streamlit 页面卡死排查

```
1. 检查是否有 widget key 冲突（控件 key vs session_state 赋值）
2. 检查 main() 是否有异常中断（try/except 吞掉错误？）
3. 清 __pycache__ + 重启 Streamlit
4. 用 ast.parse() 验证语法
5. 逐段注释定位阻塞位置
```

### 8.3 回测结果异常排查

```
1. 收益率 > 1000%？→ 检查信号重用（last_exit_idx）
2. 交易次数异常多？→ 同上
3. 入场价不等于当前K线？→ 前视偏差
4. 仓位 < $10？→ 最小仓位保护
5. 模拟数据起点离谱？→ _LAST_REAL_CLOSE 锚点
```
