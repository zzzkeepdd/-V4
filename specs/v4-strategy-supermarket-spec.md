## 1. 策略调度引擎

### 1.1 三层过滤架构

```
OKX 行情(多币种独立) → Layer 1 行情映射表 → Layer 2 逐策略门控 → Layer 3 风控闸门 → 下单
```

**Layer 1 — 行情映射表（粗筛）**

DeepSeek 收到结构化行情特征后输出每币种的行情状态，查表得候选策略池：

| 行情状态 | 候选策略 |
|----------|---------|
| TRENDING_UP | trend_following, smc_structure, bos_trailing_stop |
| TRENDING_DOWN | trend_following, smc_structure, bos_trailing_stop |
| RANGING | grid_trading, swing_range |
| HIGH_VOLATILITY | bb_breakout |
| TREND_EXHAUSTION | trend_exhaustion |

行情不互斥——BTC/ETH/SOL 各自独立判断。状态由 DeepSeek 输出，本地规则兜底（ADX>25→TRENDING, ADX<20→RANGING, ATR/ATR_MA>1.5→HIGH_VOL）。

**Layer 2 — 逐策略门控（精筛）**

候选池中每个策略执行独立激活条件检查：

| 策略 | 激活条件 |
|------|---------|
| trend_following | ADX>25 AND close>EMA20 |
| smc_structure | ADX>25 AND 检测到BOS结构 AND 回调至fib 0.5-0.618 |
| bos_trailing_stop | ADX>25 AND close>EMA50 |
| grid_trading | ADX<20 AND BB_width<2% |
| swing_range | ADX<20 AND close距区间边界<ATR |
| bb_breakout | BB_width>2% AND volume>volume_ma×1.5 |
| trend_exhaustion | 趋势运行≥20根K线 AND RSI顶/底背离 |

**Layer 3 — 风控闸门**

最多3笔持仓。同币种不双向——反向信号先平后开。超过3策略激活时按信号强度评分(SSS)排序。

### 1.2 信号强度评分 (SSS)

当多个策略同时激活同一币种时，按 SSS 择优。SSS 范围 0-100：

```
SSS = ADX贡献(0-30) + 均线对齐贡献(0-30) + 成交量确认(0-20) + 策略数量共振(0-20)

ADX贡献:     ADX>40=30分, ADX>30=20分, ADX>25=10分
均线对齐:    close>EMA20+EMA50+EMA200全在下方=30分, 两条=20分, 一条=10分
成交量确认:  volume>volume_ma×2=20分, >1.5=10分
策略共振:    3+策略同时激活=20分, 2策略=10分, 1策略=0分
```

SSS>60: 强信号，DeepSeek 放大系数可取 2-3x。SSS 30-60: 中等，允许 1-2x。SSS<30: 弱信号，仅基底或拒绝。

多策略激活同一币种时，选 SSS 最高者执行。

### 1.3 DeepSeek 行情判断 Prompt

DeepSeek 接收结构化输入（非自由文本）：

```json
{
  "coin": "BTC",
  "features": {
    "adx": 32.5,
    "close_vs_ema20": "above",
    "close_vs_ema50": "above",
    "atr_ratio": 1.2,
    "bb_width_pct": 2.1,
    "volume_ratio": 1.6,
    "trend_bars": 15,
    "rsi_divergence": false
  }
}
```

DeepSeek 输出严格 JSON：

```json
{
  "market_state": "TRENDING_UP",
  "confidence": 82,
  "reasoning": "ADX 32.5 + 价格在EMA20/50上方 + 趋势运行15根"
}
```

## 2. 双层仓位模型

### 2.1 参数定义（三个明确术语）

| 术语 | 定义 | 谁控制 |
|------|------|--------|
| **账户余额** | OKX 账户实际 USDT 余额（只读） | OKX |
| **本金上限** | 用户设定的交易资金上限，≤账户余额 | 用户滑块 |
| **可用保证金** | 本金上限 - 已占用保证金 | 系统计算 |

### 2.2 开仓公式

```
每笔投入 = 本金上限 × 仓位比例（用户滑块 1-5%，默认 2%）
名义仓位 = 每笔投入 × 杠杆倍数（用户滑块 1-20x，默认 3x） × DeepSeek放大系数（0-3x，默认 1x）
止损亏损 = 名义仓位 × 策略止损距离

硬帽检查：止损亏损 ≤ 本金上限 × 3%
硬帽通过 → 开仓
硬帽不通过 → 拒绝开仓，提示"杠杆过高或仓位过大"
```

### 2.3 DeepSeek 放大系数决策

DeepSeek 根据行情置信度+策略共振度输出放大系数 0-3：

| 条件 | 放大系数 | 说明 |
|------|---------|------|
| 3策略共振 + ADX>30 + 量能放大 | 2.5-3.0 | 顶格 |
| 2策略共振 + ADX>25 | 1.5-2.5 | 中等放大 |
| 单策略 + ADX>25 | 1.0-1.5 | 微调 |
| 行情混沌/策略冲突 | 0.5-1.0 | 保守 |
| 剧烈波动 + 方向不明 | 0 | 空仓等待 |

### 2.4 币种仓位分配（4-4-2）

当多币种同时有信号时：

```
BTC: 40% 份额
ETH: 40% 份额
SOL: 20% 份额
```

某币种无信号时，份额不补（宁可空着）。同一币种多策略激活→共享该币种份额，SSS 择优执行。

### 2.5 3% 硬帽的滑点防御

硬帽检查不是理论计算——内置 15% 滑点缓冲：

```
实际亏损估算 = 名义仓位 × 止损距离 × 1.15
检查：实际亏损估算 ≤ 本金上限 × 3%
```

## 3. 风控系统

### 3.1 硬编码规则（DeepSeek 不可改）

| 规则 | 值 |
|------|-----|
| 最大杠杆 | 20x |
| 最大持仓数 | 3 笔 |
| 单笔亏损硬帽 | ≤ 本金上限 × 3% |
| 同币种双向 | 禁止，反向信号先平后开 |
| 保证金模式 | 逐仓（每笔独立保证金） |
| 杠杆上限熔断 | 单日亏损达本金 5% → 杠杆降至 1x，手动恢复 |
| ATR 波动熔断 | ATR > ATR_MA × 3 → 暂停新开仓 |
| 交易频率 | 每策略每小时最多 3 次信号 |

### 3.2 满仓信号队列

当已有 3 笔持仓时，新信号进入队列。规则：

```
队列容量: 最多 5 个待执行信号
排序: SSS 降序
替换: 队列满时，新信号 SSS 高于队列最弱者 → 替换，否则丢弃
执行: 任一持仓平仓 → 立即从队列取 SSS 最高者执行
```

### 3.3 逐仓爆后风险传染防御

单仓位爆仓清算后，自动检查其余仓位的保证金率：

```
保证金率 < 150% → 自动减仓至保证金率 ≥ 200%
无法减仓（已达最小仓位）→ 全部平仓
```

### 3.4 熔断优先级

当多个熔断条件同时触发：

```
优先级 1 (最高): ATR波动熔断 → 暂停全部新开仓
优先级 2:        单日亏损熔断 → 杠杆降至 1x
优先级 3:        回撤熔断 → 锁定本金 80%
优先级 4:        连续失败暂停 → 对应策略暂停 1 小时
```

## 4. 数据层稳健性

### 4.1 断网保活

心跳间隔 10 秒。断网检测（30 秒无响应）→ 冻结新信号，维持已有持仓。重连后：

```
1. 从 OKX 拉取当前持仓列表（权威数据源）
2. 与本地状态对比
3. 交易所状态 wins —— 覆盖本地
4. 恢复信号生成
```

### 4.2 AI 降级链

```
Tier 1: DeepSeek reasoner (deepseek-reasoner) — 完整分析
   ↓ 超时/报错 10s
Tier 2: DeepSeek chat (deepseek-chat) — 简化 prompt
   ↓ 超时/报错 10s
Tier 3: 本地规则引擎 — ADX+EMA交叉判断行情，取候选池第一个策略
```

降级事件记录到 AI 决策日志，标记降级原因和层级。

### 4.3 CSV/WS 数据仲裁

历史回测用 CSV 缓存数据。实盘用 OKX WebSocket 实时数据。显式标记数据来源，不在运行中混用。

### 5.3 并发 UI 防冻

PyQt6 主线程只做渲染。三个工作线程：

| 线程 | 职责 | 通信方式 |
|------|------|---------|
| OKX 线程 | WebSocket 行情 + REST 订单 | Signal → 主线程 |
| AI 线程 | DeepSeek API 调用 | Signal → 主线程 |
| 策略线程 | 门控检查 + 风控 + 下单 | Signal → 主线程 |

策略线程用 QMutex 保护共享状态（持仓列表、信号队列）。

### 5.4 策略超市可视化

策略管理标签页：卡片网格布局，每张卡片显示策略名 + 标签(适用行情) + 质检评分(5星) + 标的覆盖(BTC/ETH/SOL 勾选图标) + 预置参数包摘要。点击展开参数详情面板，可切换饼查看不同币种的参数包值。

## 6. 部署与运维

### 6.1 状态持久化

每 30 秒自动保存状态快照到 `config/session_state.json`：

```json
{
  "positions": [...],
  "active_strategy": "smc_structure",
  "last_ai_decision": {...},
  "last_trade_time": "2026-05-21T14:30:00Z",
  "capital_allocated": 8000,
  "capital_remaining": 2000
}
```

### 6.2 重启恢复

```
1. 读取 session_state.json
2. 从 OKX 拉取当前持仓（权威数据源）
3. 比对——交易所状态 wins
4. 交易所无该持仓但本地有 → 删除本地记录（已被手动平仓）
5. 交易所有持仓但本地无 → 导入并设为"外部持仓"（不自动管理）
6. 恢复 AI 状态：重新判断行情 → 重新匹配策略
```

### 6.3 假死孤儿订单检测

每 60 秒检查一次：OKX 有挂单但本地状态显示"已成交"→ 标记异常，弹窗提示用户手动确认。OKX 有持仓但本地无记录 → 导入追问用户意图。

## 7. 策略文件加载规范

### 7.1 加载路径

扫描 `D:\量化平台V4\strategies\` 目录（使用 V3 的 strategies 目录）下所有 `.py` 文件，解析标签注释和预置参数包。

### 7.2 标签解析规则

每个策略文件头部 `#` 注释中提取：

```
# 适用行情: <值>
# 不适行情: <值>
# 交易频率: <值>
# 标的: <逗号分隔币种>
# 质检结果: <分数>/5
```

### 7.3 预置参数包解析

从策略文件中提取 PARAMS_BTC、PARAMS_ETH、PARAMS_SOL 字典。SOL 参数包空洞（仅含注释"待跑回测"）时，用 DEFAULT 参数包替代并标注 ⚠️。

### 7.4 质检阈值

质检分数 < 4/5 的策略不加载（volume_breakout 和 scalping_strategy 自动排除）。4/5 策略加载但标注已知风险。
